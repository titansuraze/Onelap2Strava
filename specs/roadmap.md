# 顽鹿运动 -> Strava 演进路线图

本文档记录整个产品的四阶段演进规划，作为后续迭代的"北极星"。每个 Phase 都列出目标、范围边界、关键技术点、交付标准和本阶段明确不做的事，避免提前引入复杂度。

---

## Phase 1：离线脚本打通链路（当前阶段）

### 目标
在本地跑通"顽鹿 Fit 文件 → 坐标修正 → Strava 上传"的完整链路，验证所有核心技术难点可解。

### 范围边界
- Fit 文件由用户**手动**从顽鹿 App 导出并放到本地指定目录。
- 单用户、命令行、本机运行。
- Strava 授权走一次 OAuth，token 持久化到本地 JSON。

### 关键技术点
- GCJ-02 ↔ WGS-84 双向坐标转换（迭代反解法）。
- Fit 文件解析/修改/回写（semicircles 单位处理、保留 HR/功率/踏频等字段）。
- Strava OAuth 2.0（本地 HTTPServer 接收回调）。
- Strava 上传接口 + 去重检测（external_id + 时间窗）。
- 真实数据回归测试：以"同一路线两次骑行"夹具验证 GCJ-02 偏移被消除。

### 交付标准
- `uv sync` 后一键可运行。
- `uv run onelap2strava upload <fit>` 成功返回 Strava activity URL。
- Strava 上看到的路线与顽鹿 App 中路线位置一致（在正确的道路上）。
- 有完整的 README 和夹具回归测试。

### 本阶段不做
- 顽鹿接口自动拉取。
- Web UI / 前后端分离。
- 多用户、账号体系。
- 定时任务 / Docker / 部署。
- 复杂的模糊去重（先用 external_id + 时间窗简单方案）。

---

## Phase 2：自动化顽鹿数据拉取

### 目标
移除"手动导出 Fit"这一步，实现登录顽鹿后自动获取最新骑行的 Fit 文件。

### 范围边界
- 仅对接顽鹿**私有接口**（无公开 API）。
- 仍是本地 CLI，用户提供顽鹿账号凭证（本地保存，不上云）。
- 仅拉取最新 N 条骑行记录（默认 1 条）。

### 关键技术点
- 抓包逆向：用 Charles / mitmproxy / Fiddler 抓顽鹿 App 或微信小程序 HTTPS 流量。
- 优先分析微信小程序（JS 包可反编译、接口签名多为明文）。
- 鉴权机制：识别 token/signature 算法（常见 `Authorization: Bearer` 或 `X-Sign: timestamp+md5`）。
- 关键接口：登录 / Token 刷新 / 骑行列表 / Fit 文件下载链接。
- 本地安全存储：凭证加密（`keyring` 或简易对称加密），token 持久化。

### 交付标准
- `uv run onelap2strava sync` 一条命令完成：登录顽鹿 → 拉最新 Fit → 坐标修正 → 上传 Strava。
- 顽鹿 token 过期自动重新登录或刷新。

### 本阶段不做
- 历史全量同步（只同步最新一条）。
- 密码登录以外的方式（短信验证码、扫码等等留到后续）。
- 公开分发（避免账号安全/合规风险）。

### 风险与备注
- 顽鹿私有接口随时可能改版，需要隔离"接口层"便于维护。
- 严格限制使用场景为**用户自己的账号**，避免多账号池之类的公开产品形态。

---

## Phase 3：去重与容错增强

### 目标
应对长期使用中的真实场景：避免重复上传、应对各种失败情况。

### 范围边界
- 仍是单用户 CLI 工具。
- 重点增强**同步链路**的健壮性。

### 关键技术点
- **模糊去重**：仅靠 `external_id` 不够（顽鹿可能对同一骑行重新生成导出）；增加"开始时间 ± 10 分钟 + 总时长差 < 5% + 起点距离 < 500m"的模糊匹配。
- **失败重试**：上传超时/网络失败自动重试（指数退避，最多 3 次）。
- **幂等性**：同一 Fit 多次运行不产生副作用。
- **状态跟踪**：本地 SQLite 记录每次同步的 Fit 哈希、Strava activity ID、时间戳，便于审计和回溯。
- **增量同步**：Phase 2 拉取的"最新一条"扩展为"自上次同步以来的所有新骑行"。

### 交付标准
- 连续运行 `sync` 命令多次，不会产生重复活动。
- 网络异常/上传失败能自动重试并最终成功或明确失败提示。
- 本地有清晰的同步日志/状态表可查。

### 本阶段不做
- 跨设备状态同步。
- Web UI。

---

## Phase 4：Web 化（可选，视需要）

### 目标
把工具升级为可托管的 Web 服务，支持更友好的授权体验和（可能的）多用户场景。

### 范围边界
- 核心依旧是 Phase 1-3 的同步逻辑，Web 层是封装。
- 决定是"单用户自部署"还是"公开 SaaS"前，先按单用户自部署做，避免合规和账号安全风险。

### 关键技术点
- 后端：**FastAPI**（异步、OAuth 回调简单、自动文档）。
- 任务队列：Celery + Redis（或 FastAPI `BackgroundTasks` 起步）。
- 前端：React/Vue + Tailwind 或 HTMX + Jinja2（MVP 用 HTMX 更省事）。
- 存储：SQLite（单用户）或 PostgreSQL（多用户）。
- 部署：Docker Compose，可放 Fly.io / Railway / 自有 VPS。
- 安全：Strava token 加密存储，顽鹿凭证处理需极其谨慎（明确用户协议）。

### 交付标准
- 浏览器里完成 Strava 授权和顽鹿登录。
- 可查看同步历史、手动触发同步、开关自动同步。
- 单容器 `docker compose up` 启动。

### 本阶段不做
- 大规模多租户运营（如果走这步需要重新评估合规）。
- 付费功能。
- 移动端 App。

---

## 阶段之间的依赖与取舍

```
Phase 1 (当前)
   │  打通所有技术难点
   ▼
Phase 2 ───── 可选：跳过直接做 Phase 3/4，但 UX 会差
   │  自动化顽鹿侧
   ▼
Phase 3 ───── 推荐：长期使用必备
   │  去重 + 容错
   ▼
Phase 4 ───── 可选：自己用 CLI 就够了
   Web 化
```

每完成一个 Phase，先**停下来用一段时间**再决定是否做下一阶段，避免过早优化。
